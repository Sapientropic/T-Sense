# v0.5-alpha Alert And Review Inbox Design

Status: Accepted planning boundary, updated 2026-05-09. v0.5-alpha.1 and the
v0.5-alpha.2 fast job alert hardening slice have shipped; this document remains
the product boundary while
`docs/agent-cli-contract.md` owns the shipped CLI/schema contract.

This document is the detailed planning authority for the v0.5-alpha iteration.
`ROADMAP.md` should link here and keep only a short phase summary. Do not mirror
the full rules in README files, the agent contract, or profile templates until
implementation makes those surfaces current.

## Goal

Make TG Channel Scanner useful as a repeated, profile-driven monitoring loop:

1. Each profile runs on its own schedule and alert policy.
2. Only high-priority new or changed items interrupt the user.
3. Everything else goes into a local review dashboard for batch processing.
4. Follow-up feedback can update the next LLM screening prompt with a short,
   confirmable diff.

The product shape is not "one global feed plus a generic dashboard." The v0.5
shape is "profiles as durable monitoring tasks."

## v0.5-alpha.1 Shipped Slice

The first implementation slice covers the repeatable monitor core:

- `scripts/monitor.py run --profile-id <id>` reuses scan/report, writes
  `run_manifest_v1`, creates review cards, evaluates alerts, and records
  delivery attempts.
- `.tgcs/profiles.toml` uses `profile_run_config_v1` for profile run config and
  delivery targets.
- `.tgcs/tgcs.db` stores runs, artifacts, review cards, feedback events, alert
  events, delivery targets, profile snapshots, and profile patch suggestions.
- Telegram Bot delivery is the first adapter; tests and CI use dry-run, while
  live sends require `--delivery-mode live` and `TGCS_TELEGRAM_BOT_TOKEN`.
- `dashboard/` is a Vite React app served by `scripts/dashboard_server.py` on
  `127.0.0.1`. Binding it to a non-loopback host is allowed for advanced local
  networks, but the server warns because dashboard state can include local
  workflow context and report artifacts may include raw context. It should only
  be exposed to trusted interfaces.

Still deferred from this slice: webhook/email/Saved Messages adapters, full
source management UI, hosted/team features, real-time feed semantics, and true
per-source watermarks.

## v0.5-alpha.2 Fast Job Alerts

The next slice adds a separate fast lane for job monitoring:

- `jobs-fast` is a starter profile task for Telegram developer opportunity
  channels: full-time roles, contract work, freelance gigs, and Mini Apps/TON
  project leads.
- The default cadence is 15 minutes during configured work hours.
- The scan window is 2 hours by default so delayed or missed scheduler runs do
  not create gaps.
- The interruption gate remains stricter than the scan window:
  `rating=high`, `decision_state in ["new", "changed"]`, and
  `alert_max_age_minutes=60`.
- Delivery defaults to the existing private Telegram Bot target. The target can
  remain disabled until the local chat id and `TGCS_TELEGRAM_BOT_TOKEN` are set.
- Dashboard profile controls may switch alert delivery between `work_hours`,
  `all_day`, and `muted`. This stores a local SQLite runtime override so the
  next monitor run can respect the user's interruption choice without rewriting
  profile TOML.
- Dashboard Source Yield merges review-card outcomes with the latest
  `scan.meta.json` source-health counters, so the user can see `kept/raw`,
  latest-card yield, and high-value outcomes without exposing raw Telegram
  message bodies.
- Dashboard source insights convert those stats into promote/prune/watch
  suggestions, plus conservative observe suggestions for one-off high-signal
  sources, so channel maintenance becomes an action list instead of a raw
  metrics table without over-promoting cold-start channels. Watch also covers
  latest-run sources with fresh messages but no review cards, which points to
  possible prefilter/profile mismatch before recommending pruning. Sources whose
  latest scan failed are surfaced as Access/watch items before any value
  judgment.
- Dashboard keep, skip, and false-positive decisions export back to the existing
  `tgcs-feedback-v1` JSONL import path through Settings or `tgcs feedback
  export`, without leaking private dashboard note bodies. Settings also shows
  keep/skip/false-positive counts before export so the feedback loop is not a
  blind total. Exports include sanitized rating and decision status so validation
  can measure false positives by priority and novelty.
- Inbox cards have local triage filters for all, high, new/changed, and
  low/medium cards. These filters are view-only and do not bulk-change review
  state, which keeps high-volume triage faster without increasing accidental
  action risk.
- Follow-up profile diffs require a non-empty note; the UI disables empty
  follow-up submissions and the backend rejects them, avoiding generic rules
  that would make profiles noisier over time. Pending diff cards show the
  originating review-card title, profile path, created time, and `+/-` diff
  scale before the user applies the change. Applied diffs can be reverted from
  the saved profile snapshot while the file still equals the applied proposal;
  if the user edited the profile after applying, automatic revert is refused to
  avoid overwriting manual work. Applying also verifies the profile still
  matches the base hash captured when the diff was suggested, so manual edits
  made before apply are not overwritten by an older proposal.
- Dashboard Runs project report diagnostics into the run-quality row, including
  warning/failure counts and the top diagnostic code, so incomplete scans and
  empty-result causes are visible without opening the full report.
- Dashboard empty state shows a First Useful Report checklist from
  `setup_status.checks`: profiles, source access, first dry-run, and alert
  delivery. It displays reviewable commands only; the dashboard does not execute
  them. When the latest run failed during scan with source-access diagnostics,
  source access and first run are marked blocked before delivery setup is
  suggested.
- Dashboard first screen shows `opportunity_summary` for the latest run: high
  actionable count, scanned/matched/alert counts, Top 3 sanitized review cards,
  All Clear state, latest-run decision-memory counts, or the top run diagnostic
  when source access failed. It also carries a `next_action` recommendation so
  the dashboard and agent-facing JSON agree on whether to review Inbox cards,
  keep cadence, or fix source access. Replay/offline runs that bypass scan via
  `--scan-input` reuse `scan_meta.total_messages_collected` for first-screen
  counts instead of showing a false `0/0`.
- Dashboard first screen also shows a local 14-day validation summary: run
  count, high-card count, pending count, review actions, and one validation
  next action. This is behavior evidence only; it does not include raw Telegram
  text or private follow-up note bodies.
- Review-card, alert, opportunity-summary, feedback-export, and report titles
  now share placeholder-aware display logic: model-required missing markers such
  as `Unknown` and `Not specified` remain available in detail fields, but they
  are skipped when building the primary title. Legacy cards already written to
  SQLite are recovered at read time from their sanitized `item_json`.
- Custom HTML/Markdown report cards render machine field names as human labels
  (`apply_url` becomes `Apply URL`), link URL-like fields through the safe-link
  sanitizer, and avoid duplicating `why` in both the detail grid and narrative
  body.
- Runs can open generated report artifacts through a local-only route restricted
  to workspace-local `runs/` `report.html` or `report.md` files, so the
  dashboard can link back to report context without exposing raw scan JSONL or
  becoming an arbitrary file server.
- `jobs-fast` uses a cheap local keyword prefilter before semantic extraction:
  each run scans Telegram first, filters matched messages into
  `prefiltered-scan.jsonl`, and skips the report/LLM stage entirely when no
  keyword matches.
- The jobs profile deduplicates by company + role. `apply_url` remains a
  rendered detail/link field, but is not part of the primary dedupe key because
  model extraction may add or omit it across runs for the same Telegram post.

The 24-hour daily report remains the audit lane. Fast alerts should not replace
daily review, source traceability, or raw context expansion.

## Target User

The first target is a personal power user or self-hosted open-source user who
monitors multiple Telegram topics and can forget to revisit important items.

ADHD-friendly defaults matter:

- Prefer one obvious next action over a wall of settings.
- Turn `follow_up` into a prompt-edit suggestion immediately, before the user
  forgets the context.
- Keep changes reversible through profile snapshots and diff review.
- Do not hide useful automation behind enterprise-style approval flows.

## Product Principles

1. Profile owns the monitoring policy.
   Schedule, alert rules, delivery targets, prompt adjustments, and source topic
   filters should be profile-level settings. Global defaults are only fallbacks.

2. Alerts are interruptions, not reports.
   Default alert condition is `rating=high` and
   `decision_state in ["new", "changed"]`. Profile rules may tighten or broaden
   this, but low and medium items should not interrupt by default.

3. Dashboard is the review home.
   The dashboard exists to clear pending cards, edit profiles, inspect recent
   runs, and recover from failures. It should not become a generic analytics
   dashboard in v0.5-alpha.

4. Low friction beats theoretical safety.
   The tool remains local-first and explicit about risky uploads, but local
   personal workflow should not require unnecessary approval steps. Prompt
   changes should be confirmable, not buried.

5. JSON contracts remain stable.
   The internal dashboard state may use SQLite, but agent-facing command output
   and import/export surfaces should stay JSON-based.

### High-frequency cost controls

The first shipped control is local prefiltering. It is intentionally only a
candidate gate, not a score: keyword matches may enter semantic extraction, but
the alert decision still depends on profile extraction, `rating=high`,
`decision_state`, and freshness.

For 5-15 minute loops, the preferred next optimization is cache-aware semantic
batching:

- Keep the stable prompt prefix byte-for-byte identical across runs: profile
  rules, output schema, examples, and extraction instructions first.
- Put only the variable incremental Telegram messages at the end.
- Use a stable provider cache key per profile lane when the provider supports
  it, for example `tgcs:jobs-fast:v1`.
- Record cached-token usage, latency, and semantic-stage cost in the run
  manifest before tuning cadence.

[⚠️ 需确认] Exact prompt-cache availability, retention, and price discounts are
provider/model-specific and must be checked against the current provider docs
before shipping provider-specific flags.

Shipped cache-aware defaults:

- `report.py` defaults to `deepseek-v4-flash` when `OPENAI_API_KEY` is absent
  and `DEEPSEEK_API_KEY` is present, even if MiniMax is also configured.
- DeepSeek V4 extraction disables thinking mode for the fast lane.
- `report.py` can explicitly select `MiniMax-M2.7`; both token-plan keys and
  standard MiniMax API keys use official OpenAI-compatible endpoints. Token
  Plan keys default to `https://api.minimaxi.com/v1`, while standard platform
  keys default to `https://api.minimax.io/v1`. `MINIMAX_BASE_URL` can override
  both defaults.
- MiniMax extraction requests split reasoning out of `message.content` and use a
  provider-valid near-deterministic temperature because MiniMax rejects
  `temperature=0`.
- LLM calls request JSON object output and record `usage`,
  `prompt_cache_hit_tokens`, `prompt_cache_miss_tokens`, `cache.hit_rate`,
  `latency_ms`, and `prompt_prefix_hash`.
- The jobs profile uses a compact `items` schema so output tokens do not dominate
  the cached-input savings.
- `jobs-fast` caps the interrupt-lane report command at
  `semantic_max_messages=20` and `semantic_max_tokens=2000`.
- `scripts/eval_deepseek_cache.py` runs local provider/cache evals without
  copying raw Telegram text into the result artifact. Workspace-local input
  paths are stored as relative paths; external input paths are reduced to file
  names, with a short hash suffix only when duplicate basenames would collide.

Recommendation from this snapshot: keep Flash as the default high-frequency
extractor, keep fast-lane batches around 20 messages, and set the output cap to
2000 tokens rather than 1200. Reserve Pro for fallback or periodic review:
within this local sample it was more conservative but not clearly better for
interrupt routing, and was usually slower. Cache makes input cost attractive,
but output tokens still dominate latency when the model emits several structured
items. [⚠️ 需确认] This history did not contain a labeled high-value job lead,
so true recall on urgent opportunities still needs live review.

The 2026-05-09 10-message comparison did not change the default route:
MiniMax M2.7 was faster and used fewer completion tokens on that run, but
returned zero items while DeepSeek Flash returned three. Keep MiniMax available
for explicit comparison/fallback until live labeled opportunity data proves its
recall is sufficient.

The later 2026-05-09 five-message Token Plan comparison also supports the same
default: DeepSeek Flash returned more items, found one high item, used fewer
completion tokens, and was faster than MiniMax M2.7 on that small sample.
[⚠️ 需确认] This remains a provider-routing smoke rather than a stable recall
benchmark because it uses only five local messages and no independently labeled

`MINIMAX_BASE_URL` only when an account needs an explicit override.
[⚠️ 需确认] In the same five-message provider smoke, MiniMax used many more
completion tokens and was materially slower than DeepSeek Flash; this is still a
provider-routing/JSON smoke, not a model-quality verdict.

Scan metadata is now reused in the Source Yield panel through source-level
`raw_count`, `kept_count`, latest-card counts, and card yield. Further reuse
should focus on:

- dynamic channel cadence: scan valuable channels more often and dormant/noisy
  channels less often;
- channel pruning candidates: surface channels with repeated raw messages but
  no high or medium items;
- keyword evolution: propose added/removed prefilter terms from false positives,
  misses, and matched keyword counts;
- budget allocation: spend semantic extraction only on profiles/channels with
  recent yield;
- freshness monitoring: detect channels whose newest message is lagging or
  whose high-value posts arrive outside the current alert window;
- source-health alerts: distinguish access failures, incomplete scans, OCR
  volume, and empty channels from true low value.

### Run Manifests

Every run should write a manifest with enough information to debug repeated
automation:

- `run_id`
- started/completed timestamps
- profile id and profile hash
- source registry hash
- scan window and source filters
- generated artifact paths
- report status, alert count, and error summary
- delivery attempts and delivery result ids when available

Runs must not silently overwrite each other. A `latest` pointer is fine, but
previous run artifacts should remain addressable.

### Review Inbox

The dashboard should create pending cards from report items.

Initial card state:

- `pending`
- `kept`
- `skipped`
- `false_positive`
- `follow_up`

Initial actions:

- `keep`
- `skip`
- `false_positive`
- `follow_up`

Later roadmap candidates:

- `snooze`
- `mute similar`
- bulk actions
- per-source or per-topic profile overrides

The review queue should suppress repeated interruptions for already handled
items. A handled item may still appear in report history.

### Alert Engine

Default alert predicate:

```text
rating == "high" and decision_state.status in ["new", "changed"]
```

Profile-level `## Alert Rules` may add conditions such as:

- deadline or time sensitivity
- source priority
- explicit keywords
- item fields from the profile extraction schema
- follow-up preference rules accepted by the user

For `jobs-fast`, `alert_max_age_minutes=60` is part of the default rule. This
keeps a wider 2-hour scan window for fault tolerance while preventing old job
posts from interrupting the user.

Alert suppression is status-aware. A sent `new` alert for a card suppresses
later repeated `new` delivery, but the same card can alert again when decision
memory marks it `changed` and the source freshness window still passes.

Alerts should include:

- concise title
- rating and decision state
- why it matched
- source refs
- report/dashboard link
- delivery timestamp

Alerts should not include the full raw Telegram message by default. The user can
open the local dashboard or report for raw context.

### Delivery Adapters

v0.5-alpha delivery targets:

1. Default: private Telegram bot.
2. Optional: webhook.
3. Optional: email.
4. Optional: Telegram Saved Messages.

Telegram bot delivery is the default because it matches the user's existing
Telegram workflow and does not require the scanner to impersonate the user for
outbound messages.

Saved Messages delivery should be treated as a separate adapter from bot
delivery. [⚠️ 需确认] It likely requires the user's MTProto session rather than
the Bot API, so implementation must re-check Telegram behavior and risk before
shipping.

### Follow-up To Profile Diff

`follow_up` is a core ADHD-friendly loop, not an optional polish feature.

Flow:

1. User clicks `follow_up` on a card.
2. Dashboard saves a local follow-up note.
3. Dashboard immediately asks whether this should affect future screening.
4. If yes, the system proposes a profile diff.
5. User reviews the diff and applies it with one action.
6. The profile snapshot changes, and the next scan uses the updated profile.
7. If the applied rule is wrong, the user can revert from the saved snapshot as
   long as the profile file has not been edited manually since apply.

Suggested writable sections:

- `## Alert Rules`
- `## Feedback Rules`
- `## Follow-up Preferences`

Do not silently rewrite a profile. Do not require manual prompt surgery for the
normal case.

Follow-up note bodies may be stored locally for the user's workflow. They should
not be inserted into future LLM prompts unless the user confirms the generated
profile diff.

### Profile Studio

Dashboard v1 should focus on profile work:

- choose active profile
- edit the local profile copy
- preview extraction prompt impact
- view accepted feedback-derived rules
- show pending profile diff
- apply/revert profile diff
- view profile snapshots

The profile templates remain starter content. User-edited profiles belong in
`.tgcs/profiles/`.

### Dashboard V1

The local dashboard should listen on `127.0.0.1` by default. Non-loopback
bindings should be explicit and noisy because dashboard state carries local
workflow context and report artifacts can contain raw context even though
SQLite/dashboard state avoids raw message storage.

Primary views:

- `Inbox`: pending cards and card actions.
- `Profiles`: edit profiles, review diffs, preview prompts.
- `Runs`: recent runs, status, report links, alert counts.
- `Settings`: delivery targets and profile schedules.

Minimal support only:

- `Sources`: do not build a full source management console in v0.5-alpha.
  Surface only failures, noisy sources, and high-value sources when they help a
  review decision.

Dashboard design should favor low-friction daily use over broad configuration.

## Internal State

Use SQLite internally when implementing the dashboard:

- `.tgcs/tgcs.db`

Candidate tables:

- `profiles`
- `profile_snapshots`
- `runs`
- `run_artifacts`
- `review_cards`
- `feedback_events`
- `profile_patch_suggestions`
- `alert_events`
- `delivery_targets`
- `source_health_snapshots`

The exact schema should be designed during implementation. This document fixes
the product boundaries, not column names.

Keep external compatibility:

- Existing CLI JSON envelopes remain the agent-facing API.
- Existing report feedback JSONL remains importable.
- Existing `item-memory.json` can be bridged or migrated later.

## Privacy And Risk Boundaries

Always keep out of SQLite and docs:

- Telegram session strings
- API keys
- raw credentials
- Bot tokens

Prefer storing raw Telegram message bodies in run artifacts and reports rather
than duplicating them into central long-lived state. The dashboard can resolve
raw context through source refs and artifact paths.

Cloud LLM, OCR, STT, and full-video processing remain explicit settings.

## Out Of Scope For v0.5-alpha

- Hosted SaaS.
- Multi-user permissions.
- Team dashboards.
- Generic analytics dashboarding.
- Full source management UI.
- Real-time streaming feed.
- Automatic profile rewriting without user diff review.
- Per-source or per-topic profile binding beyond roadmap/design notes.

## Roadmap Follow-ups

Move these to later phases unless implementation proves they are needed earlier:

- Per-source or per-topic profile binding.
- `snooze` and `mute similar` review actions.
- Rich source-health trend analysis.
- Team-safe profile sharing.
- Hosted deployment.
- True per-source scan watermarks.
- Mobile-first dashboard or push notification UX.

## Exit Criteria

v0.5-alpha is complete when:

- A user can configure at least two profiles with different schedules.
- Repeated runs do not overwrite prior reports and can be traced through run
  manifests.
- High-priority new or changed items can trigger private Telegram bot delivery.
- Non-alerted items remain reviewable in the dashboard.
- Inbox actions prevent repeated interruptions for handled items.
- `follow_up` can produce a profile diff, apply it to the local profile copy,
  and safely revert it from the saved pre-apply snapshot.
- The dashboard works locally on `127.0.0.1`.
- Agent JSON contracts remain usable without the dashboard.
